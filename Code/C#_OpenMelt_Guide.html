A dual-mode spinning combat robot with RC control, IMU sensing, and phase-aligned motor control

What is Spinlayden?

Spinlayden is a combat robot built for competitions like BattleBots. What makes it special? It can spin its entire body like a top while still moving in any direction you want! This is called a "MeltyBrain" robot.

Think about it: when something is spinning really fast, how do you make it move left or right? That's the challenge we're solving! The robot uses a Raspberry Pi computer running at 100 times per second to figure out exactly when to pulse the motors to create movement.

Two Operating Modes

directions_car Drive Mode

Simple forward-only driving, like a car. Use the left stick to go forward, right stick to steer left/right. This is the safe starting mode when you first power on.

psychology MeltyBrain Mode

Full-body spinning! The robot spins like a top, but you can still control which direction it moves using the right stick. Left stick controls how fast it spins.

The Control Loop (100 times per second!)

Here's what happens every 0.01 seconds (that's 10 milliseconds, or 1/100th of a second):

1
Read RC Controller — What are the sticks doing? Is the kill switch on?
2
Read Sensors — Which direction is the robot facing? How fast is it spinning?
3
Calculate True Heading — Combine gyroscope (fast) with compass (accurate) to know exactly which way we're pointing
4
Check Safety — Is the kill switch active? Which mode are we in?
5
Calculate Motor Powers — Based on the mode, figure out how much power each wheel needs
6
Send to Motors — Tell the ESCs (motor controllers) what to do

Reading the RC Controller

Your RC transmitter (the controller) sends signals to a receiver on the robot. The receiver outputs these as PWM pulses on different channels. Think of PWM like a light switch that turns on and off really fast - the longer it's "on", the higher the value.

What is PWM?

PWM = Pulse Width Modulation. Imagine you have a light switch that turns on and off 50 times per second. If it's on for 1 millisecond (1ms), that's 0%. If it's on for 2ms, that's 100%. If it's on for 1.5ms, that's 50%.

RC receivers work the same way: they send pulses between 1000µs (1ms) and 2000µs (2ms). The Raspberry Pi measures how long each pulse is to figure out where your sticks are.

The 6 Channels We're Reading

Channel GPIO Pin What It Controls Output Range
Ch1 GPIO 5 Right Stick Left/Right -1.0 (left) to +1.0 (right), 0 = center
Ch2 GPIO 6 Right Stick Up/Down -1.0 (down) to +1.0 (up), 0 = center
Ch3 GPIO 16 Left Stick Up/Down (Throttle) 0.0 (bottom) to 1.0 (top)
Ch5 GPIO 17 (Not Used) 0.0 to 1.0
Ch6 GPIO 22 Kill Switch (Safety) 0.0 (off) or 1.0 (on)
Ch7 GPIO 27 Mode Switch 0.0 (MeltyBrain) or 1.0 (Drive)

How We Measure the Pulses (Edge Detection)

The Raspberry Pi can detect when a GPIO pin goes from LOW (0 volts) to HIGH (3.3 volts) and vice versa. These are called edges:

  • Rising Edge: Signal goes from LOW → HIGH (start of pulse)
  • Falling Edge: Signal goes from HIGH → LOW (end of pulse)

We record the time at the rising edge, then calculate the difference at the falling edge. That's the pulse width!

code GpioRcInputReader.cs - Measuring Pulse Width 
// This function is called every time a GPIO pin changes (rises or falls)
private void OnPinValueChanged(int channelIndex, PinValueChangedEventArgs args)
{
    // Get the current timestamp
    var now = Stopwatch.GetTimestamp();

    if (args.ChangeType == PinEventTypes.Rising)
    {
        // Rising edge: Record when the pulse started
        _risingTimestamps[channelIndex] = now;
    }
    else
    {
        // Falling edge: Calculate how long the pulse was
        long risingTime = _risingTimestamps[channelIndex];

        // Convert from CPU ticks to microseconds (µs)
        float elapsedUs = (float)(now - risingTime) * 1_000_000f / Stopwatch.Frequency;

        // Smooth the value using EMA (Exponential Moving Average)
        // This removes jitter/noise from the measurements
        float alpha = SmoothingAlpha[channelIndex];
        _pulseWidthsUs[channelIndex] =
            _pulseWidthsUs[channelIndex] * (1 - alpha) + elapsedUs * alpha;

        // Track the kill switch to know if RC signal is alive
        if (channelIndex == 4) // Channel 6 (index 4)
            _lastKillSwitchPulseTimestamp = now;
    }
}

warning Safety Check: Signal Validity

If we don't receive a pulse on the kill switch channel for more than 100 milliseconds, we assume the RC signal is lost (maybe you walked too far away, or batteries died). The robot immediately stops all motors for safety!

Reading the IMU Sensor (WT901)

IMU = Inertial Measurement Unit. It's like the motion sensor in your phone that knows when you tilt or rotate it. Our WT901 sensor has three types of sensors built in:

  • Accelerometer: Measures acceleration (like feeling yourself pushed back in a car)
  • Gyroscope: Measures rotation speed (how fast you're spinning)
  • Magnetometer: Measures magnetic field (like a compass) - we turn this OFF because motors create magnetic interference

Why Do We Need Sensors?

For MeltyBrain mode to work, we need to know exactly which direction the robot is facing at all times while it's spinning. If we don't know the heading, we can't time the motor pulses correctly, and the robot just spins in place instead of moving!

How the Sensor Talks to the Pi (UART Serial)

The WT901 sensor sends data over a serial connection at 9600 baud (9600 bits per second). It sends small packets of data:

  • Packet Type 0x51: Acceleration data (ax, ay, az)
  • Packet Type 0x52: Angular velocity data (how fast it's spinning in each direction)
  • Packet Type 0x53: Angle data (roll, pitch, yaw - which direction it's facing)

Configuring for High-Speed Spinning

When we start up, we send special commands to configure the sensor for a spinning robot:

code Wt901SensorReader.cs - Sensor Configuration 
// Configure WT901 for spinning platform
byte[] unlock    = [0xFF, 0xAA, 0x69, 0x88, 0xB5]; // Unlock configuration
byte[] axis6     = [0xFF, 0xAA, 0x24, 0x01, 0x00]; // Use 6-axis mode (no magnetometer)
byte[] filtk     = [0xFF, 0xAA, 0x25, 0x05, 0x00]; // Less smoothing = faster response
byte[] bandwidth = [0xFF, 0xAA, 0x1F, 0x00, 0x00]; // Maximum bandwidth (256 Hz)
byte[] save      = [0xFF, 0xAA, 0x00, 0x00, 0x00]; // Save settings

// Send each command with a small delay between them
_serialPort.Write(unlock,    0, 5); Thread.Sleep(10);
_serialPort.Write(axis6,     0, 5); Thread.Sleep(10);
_serialPort.Write(filtk,     0, 5); Thread.Sleep(10);
_serialPort.Write(bandwidth, 0, 5); Thread.Sleep(10);
_serialPort.Write(save,      0, 5);

Why 6-Axis Mode?

The magnetometer (compass) gets confused by the motors' magnetic fields. In 6-axis mode, the sensor only uses the gyroscope and accelerometer, which are more accurate for a spinning robot. The tradeoff? It can drift over long periods, so we use a complementary filter in Step 3 to fix that!

3

Calculating True Heading (Complementary Filter)

We have two ways to know which direction the robot is facing, but each has problems:

  • Gyroscope: Super fast and accurate for short periods, but slowly "drifts" over time (errors add up)
  • AHRS (Attitude Heading Reference System): Slow to respond but doesn't drift over time

A complementary filter combines both! It uses the gyroscope for fast response and the AHRS to slowly correct any drift.

How It Works (Every 10ms)

  1. Integrate the gyroscope: Add (angular velocity × time) to our heading estimate
  2. Calculate error: How different is our estimate from what the AHRS says?
  3. Blend in correction: Add a tiny bit (0.2%) of that error back to slowly fix drift
  4. Wrap angles: Keep everything between -180° and +180°
code BrainControlLoop.cs - Complementary Filter 
// Step 1: Integrate gyroscope (fast, responsive)
// LoopDt = 0.01 seconds (our loop runs 100 times per second)
_lastHeading += sensorData.CurrentSpin * LoopDt;

// Step 2: Calculate error from AHRS
float ahrsError = sensorData.Heading - _lastHeading;

// Step 3: Handle angle wrap-around
// If error is > 180°, we went the "wrong way" around the circle
if (ahrsError >  180f) ahrsError -= 360f;
if (ahrsError < -180f) ahrsError += 360f;

// Step 4: Blend 0.2% of AHRS error for drift correction
// AhrsCorrAlpha = 0.002 (that's 0.2%)
_lastHeading += ahrsError * AhrsCorrAlpha;

// Step 5: Wrap heading to -180° to +180°
if (_lastHeading >  180f) _lastHeading -= 360f;
if (_lastHeading < -180f) _lastHeading += 360f;

Example: Why This Works

Imagine you're spinning and your gyro says you're at 45°, but AHRS says you're at 47°. There's a 2° error. Instead of instantly jumping to 47° (which would be jerky), we add 0.2% of that 2° error = 0.004°.

Over 500 loops, that 0.004° adds up to fix the full 2° drift, but it's so gradual you don't notice! Meanwhile, the gyro keeps giving us super fast, smooth updates every 10ms.

4

Safety Interlock System

Before we do ANYTHING with the motors, we check two critical things:

  1. Is the kill switch ON? If not, motors stay off no matter what.
  2. Is the RC signal valid? If we lost connection, motors stay off.

We also use this step to determine which drive mode we're in based on the mode switch position.

code SafetyInterlock.cs - Safety Check 
public SafetyState Evaluate(RcInputData rcInput)
{
    // Kill if RC signal is lost OR kill switch is off
    // Kill switch must be >= 0.8 (pretty close to 1.0) to be considered "on"
    bool isKilled = !rcInput.IsSignalValid || rcInput.KillSwitch < 0.8f;

    // Mode switch: >= 0.5 = Drive mode, < 0.5 = MeltyBrain mode
    var mode = rcInput.ModeSwitch >= 0.5f
        ? DriveMode.Drive
        : DriveMode.MeltyBrain;

    // Log when state changes (for debugging)
    if (isKilled && !_wasKilled)
        _logger.LogDebug("Kill switch ON — motors disabled");
    else if (!isKilled && _wasKilled)
        _logger.LogDebug("Kill switch OFF — armed in {Mode} mode", mode);

    _wasKilled = isKilled;
    return new SafetyState(isKilled, mode, DateTimeOffset.UtcNow);
}

warning Multiple Layers of Safety

  • Robot always starts in killed state when powered on
  • Default mode is Drive (safer than MeltyBrain)
  • If RC signal is lost for >100ms, immediate emergency stop
  • Kill switch must be firmly "on" (>0.8) to arm motors
  • You can switch modes while killed (safe to test before arming)
5

Calculating Motor Commands

Based on which mode we're in, we calculate how much power to send to the left and right motors. The two modes work completely differently!

directions_car Drive Mode

Drive mode is simple: it's like driving a car or tank. Left stick up/down controls speed, right stick left/right controls steering.

  • Left Stick Y: Forward throttle (0.0 = stopped, 1.0 = full speed forward)
  • Right Stick X: Steering (-1.0 = turn left, +1.0 = turn right)

How steering works: When you turn right, the left wheel speeds up and right wheel slows down (differential drive). This makes the robot turn!

code BrainControlLoop.cs - Drive Mode 
// Map throttle stick (0.0-1.0) to ESC range (0.5-1.0)
// ESCs treat 0.5 as "stopped" and 1.0 as "full forward"
float baseThrottle = rcInput.Throttle * 0.5f + 0.5f;
float steer = rcInput.Aileron;  // -1.0 to +1.0

// Differential drive: steer right → left wheel goes faster
float leftPower  = Math.Clamp(baseThrottle + steer * 0.5f, 0f, 1f);
float rightPower = Math.Clamp(baseThrottle - steer * 0.5f, 0f, 1f);

Example Calculation

Situation: Throttle at 50%, steering full right
baseThrottle = 0.5 * 0.5 + 0.5 = 0.75 (halfway between stopped and full)
steer = +1.0 (full right)
leftPower = 0.75 + (1.0 * 0.5) = 1.25 → clamped to 1.0 (full speed!)
rightPower = 0.75 - (1.0 * 0.5) = 0.25 (slow)
Result: Robot turns right while moving forward!

psychology MeltyBrain Mode

This is where it gets cool! The entire robot is spinning like a top. To make it move in a direction while spinning, we pulse the motors at specific moments based on where the robot is pointing.

  • Right Stick X/Y: Which direction you want to move
  • Left Stick Y: How fast to spin (0.0 = no spin, 1.0 = max spin)

The Key Insight: Phase-Aligned Control

Imagine the robot is spinning clockwise and you want it to move to the right:

  1. When the robot points right (your target), give motors more power
  2. When the robot points left (opposite), give motors less power
  3. When the robot points up or down (90° off), give motors normal power

Over one full rotation, the robot pushes harder in the right direction than any other direction. The net result? It moves right while spinning!

code TranslationEngine.cs - MeltyBrain Logic 
// Calculate which direction the pilot wants to go from stick position
var stickMagnitude = new Vector2(
    pilotCommand.TranslationX,
    pilotCommand.TranslationY
).Length();

// Calculate target heading from stick direction
if (stickMagnitude > 0.01f)
{
    targetHeading = MathF.Atan2(
        pilotCommand.TranslationY,
        pilotCommand.TranslationX
    );
}

// Convert robot's current heading from degrees to radians
float headingRad = sensorData.Heading * MathF.PI / 180f;

// Calculate heading error (how far off are we from target?)
float headingError = NormalizeAngle(headingRad - targetHeading);

// Use cosine to create sinusoidal modulation:
//   cos(0°) = +1   → max forward when pointing at target
//   cos(90°) = 0   → no correction when perpendicular
//   cos(180°) = -1 → max reverse when pointing away
float correction = stickMagnitude * 0.8f * MathF.Cos(headingError);

// Add/subtract correction from base spin speed
float leftPower  = spinSpeed + correction;
float rightPower = spinSpeed - correction;

Why Cosine?

The cosine function gives us exactly what we need! When the error is 0° (pointing at target), cos(0°) = 1 (maximum correction). When error is 90° (perpendicular), cos(90°) = 0 (no correction). When error is 180° (opposite direction), cos(180°) = -1 (reverse correction). This creates a smooth sine wave of motor power as the robot spins!

Sending Signals to the Motors (ESCs)

ESC = Electronic Speed Controller. Think of it like a smart power adapter for the motor. You tell it what speed you want (via a signal), and it controls the power going to the motor.

Speaking the ESC's Language: PWM Signals

ESCs understand the same RC servo protocol we talked about in Step 1! They expect a 50Hz signal (50 pulses per second) where the pulse width tells them what to do:

Pulse Width Power Value Motor Response
1000µs (1ms) 0.0 Full Reverse (or stopped if ESC doesn't support reverse)
1500µs (1.5ms) 0.5 Neutral (stopped)
2000µs (2ms) 1.0 Full Forward

Using Hardware PWM

The Raspberry Pi has hardware PWM, which means a dedicated chip generates these pulses with perfect timing. This is much better than software PWM (which can be jerky or inaccurate).

We configure two PWM channels in /boot/firmware/config.txt: 

dtoverlay=pwm-2chan,pin=12,func=4,pin2=13,func2=4

This sets up:

  • GPIO 12: PWM channel 0 (left motor)
  • GPIO 13: PWM channel 1 (right motor)

Converting Power to PWM Duty Cycle

The C# PWM library uses duty cycle (0.0 to 1.0) instead of pulse widths. Duty cycle is "what percentage of the time is the signal HIGH?"

Our PWM period is 20ms (50Hz). So:

  • 1000µs out of 20000µs = 0.05 (5% duty cycle)
  • 1500µs out of 20000µs = 0.075 (7.5% duty cycle)
  • 2000µs out of 20000µs = 0.10 (10% duty cycle)
code MotorController.cs - Converting to PWM 
private static double PowerToDutyCycle(float power)
{
    // Clamp power to 0.0-1.0 range (our ESCs can't go reverse)
    var clamped = Math.Clamp(power, 0f, 1f);

    // Map 0.0-1.0 power to 0.05-0.10 duty cycle
    // Formula: minDuty + (power * (maxDuty - minDuty))
    return 0.05 + (clamped * 0.05);
}

public Task SendCommandAsync(MotorCommand command, CancellationToken ct)
{
    // Convert both motor powers to duty cycles
    var leftDuty = PowerToDutyCycle(command.LeftWheelPower);
    var rightDuty = PowerToDutyCycle(command.RightWheelPower);

    // Set the hardware PWM duty cycles - happens instantly!
    _leftEsc!.DutyCycle = leftDuty;
    _rightEsc!.DutyCycle = rightDuty;

    return Task.CompletedTask;
}

Why Hardware PWM is Amazing

Once we set the duty cycle, the hardware chip keeps generating perfect pulses in the background. Even if our program gets busy doing calculations, the PWM signals stay perfectly timed at 50Hz. ESCs are very picky about timing, so this is crucial for smooth, reliable motor control!

Deploying to the Raspberry Pi

To run this on the robot, you need to compile it for the Raspberry Pi's processor (ARM64) and copy it over: 

# On your computer (Windows/Mac/Linux)
dotnet publish -c Release -r linux-arm64 --self-contained

# Copy to Raspberry Pi using SSH
scp -r bin/Release/net10.0/linux-arm64/publish/* pi@raspberrypi:~/spinlayden/

# On the Raspberry Pi, run with sudo (needed for GPIO access)
cd ~/spinlayden
sudo ./Spinlayden.Brain
